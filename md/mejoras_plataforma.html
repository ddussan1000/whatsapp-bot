Arquitectura
Los dos problemas estructurales que hay que resolver antes de escalar.
⚙️
P1 CRÍTICO ARQUITECTURA
Unificar la configuración del bot — una sola fuente de verdad
Hoy coexisten botConfig en memoria del proceso y product.system_prompt en DB. Son dos fuentes distintas para el mismo concepto.
▾
Problema actual

/api/config/bot guarda en un objeto en memoria del proceso. Si el servidor se reinicia, la configuración se pierde. Además, flows.ts ya usa product.system_prompt, así que hay dos prompts posibles y no está claro cuál gana.

Solución

Eliminar botConfig en memoria. Mover toda la configuración del bot a products (o a una tabla bot_config por org si se necesita config global). El endpoint /api/config/bot debe leer y escribir en DB, no en memoria.

Cambio concreto en backend
// ANTES — botConfig volátil en memoria let botConfig = { systemPrompt: '', keywords: [] }; app.put('/api/config/bot', (c) => { botConfig = await c.req.json(); // 💀 se pierde al reiniciar }); // DESPUÉS — persiste en DB por organización app.put('/api/config/bot', async (c) => { const { systemPrompt, keywords } = await c.req.json(); await supabase.from('organizations') .update({ bot_config: { systemPrompt, keywords } }) .eq('id', ctx.organizationId); return c.json({ ok: true }); }); // En flows.ts — único punto de lectura const prompt = product?.system_prompt ?? org?.bot_config?.systemPrompt ?? DEFAULT_PROMPT;
1
Agregar columna bot_config jsonb a organizations

O crear tabla separada org_bot_config si prefieres más estructura.

2
Migrar endpoint PUT /api/config/bot

Leer/escribir de DB en vez de variable en memoria.

3
Definir jerarquía de prompts en flows.ts

product.system_prompt tiene prioridad → luego org.bot_config.systemPrompt → luego default hardcodeado.

4
Eliminar la variable botConfig global

Borrar el objeto en memoria y todos sus usos directos.

P1 CRÍTICO ARQUITECTURA
Crear producto → el bot funciona sin pasos adicionales
Hoy hay 5 pasos obligatorios para tener un bot funcional. Deberían ser 1 (crear el producto), con el resto siendo opcional y avanzado.
▾
Flujo actual (5 pasos obligatorios)

1. Crear producto → 2. Seleccionar activo → 3. Crear campaña ligada → 4. Activar campaña → 5. Crear versión de flujo. Si falta el paso 3 o 4, el bot no responde y el error es silencioso.

Flujo objetivo (1 paso mínimo)

Crear producto con system_prompt y keywords → bot responde. Campaña, flujos y plantillas son capas opcionales para quienes quieren más control.

⚠️
Esta mejora requiere coordinación con la mejora #3 (resolver producto sin campaña) y la #6 (auto-crear campaña). Las tres juntas resuelven el problema completo.
1
El webhook funciona con solo product_id (sin campaign_id)

Ver mejora #3 para el cambio concreto en el resolver.

2
Al crear un producto, auto-crear la campaña asociada en el backend

Ver mejora #6. El usuario no tiene que ir a otra pantalla.

3
La campaña se auto-activa si el producto está activo

Sincronizar product.is_active con campaign.status en el trigger o en el handler del endpoint.

4
Flujos y plantillas permanecen como opcional avanzado

Documentar claramente en la UI que son para personalización extra, no requisito mínimo.

Wizard de configuración inicial — 5 pasos guiados
Un cliente nuevo con acceso al dashboard no sabe qué tiene que hacer primero. Un wizard lo lleva de cero a bot activo sin ambigüedad.
▾
✓
Este wizard solo aplica en el dashboard del cliente (tenant), no en el panel de plataforma. El platform admin sigue creando org y allowlist manualmente.
1
Conectar número de WhatsApp

Pedir phone_number_id y meta_token. Mostrar instrucciones de cómo obtenerlos desde Meta for Developers con capturas de pantalla. Validar que el número recibe el webhook antes de continuar.

2
Crear tu primer producto

Formulario con nombre, descripción breve y precio (o rango). El slug se genera automático. Keywords se pueden agregar después.

3
Configurar el bot

Un textarea grande con el system_prompt y un helper text explicando cómo escribirlo. Ejemplo pre-llenado según el tipo de negocio seleccionado.

4

Número de Nequi, cuenta Bancolombia, etc. Se guardan en el system_prompt o en un campo estructurado del producto.

5
Prueba de envío

P1 CRÍTICO BOT
El bot resuelve el prompt sin necesitar campaña
El webhook hoy falla silenciosamente si el producto no tiene campaña. El bot debería poder responder usando directamente el system_prompt del producto.
▾
Cambio en flows.ts — resolución en cascada
// flows.ts — antes de llamar a Haiku async function resolvePrompt(ctx: FlowContext): Promise<string> { // 1. system_prompt del producto (máxima prioridad) if (ctx.product?.system_prompt) { return ctx.product.system_prompt; } // 2. system_prompt de la campaña (si existe) if (ctx.campaign?.system_prompt) { return ctx.campaign.system_prompt; } // 3. config global de la org (nuevo, ver mejora #1) if (ctx.org?.bot_config?.systemPrompt) { return ctx.org.bot_config.systemPrompt; } // 4. Fallback hardcodeado — nunca falla return `Eres el asistente de ${ctx.org.name}. Responde en español, sé amigable y conciso.`; } // El bot NUNCA debe quedar mudo por falta de configuración
✓
Este es el cambio de menor esfuerzo y mayor impacto inmediato. Solo modifica la función de resolución en flows.ts — no toca el webhook ni la DB.
🪲
P2 IMPORTANTE BOT / DX
Errores descriptivos — no más fallos silenciosos
Cuando el bot no puede resolver producto o campaña, el log actual no da contexto suficiente para depurar. Se necesitan errores accionables.
▾
Logging estructurado en el webhook handler
// handler.ts — reemplazar warnings genéricos // ❌ Antes if (!instance) { log.warn('No instance'); return; } // ✅ Después if (!instance) { log.warn({ phone_number_id: phoneNumberId, event: 'webhook.instance_not_found', action: 'ignored', hint: 'Crear instancia en /instances con este phone_number_id' }, 'Webhook ignorado: instancia no encontrada'); return; } if (!product) { log.warn({ org_id: orgId, text: messageText, event: 'bot.product_not_resolved', hint: 'Ningún producto activo tiene keywords que coincidan' }, 'Producto no resuelto — usando fallback'); // Continuar con fallback, NO silenciar }
1
Agregar campo event a todos los logs del webhook

Formato: webhook.<acción>, bot.<acción>, ocr.<acción>. Facilita filtrar en Railway logs.

2
Nunca retornar sin loggear — especialmente en los return tempranos

Cada if (!x) return debe tener su log con hint de qué configurar.

3
Exponer últimos errores en GET /api/bot/health

Los últimos 5 warnings del webhook, accesibles desde el dashboard sin entrar a Railway.

P3 UX BOT
CTWA (Click to WhatsApp) como módulo opcional
La detección por ctwa_clid está en el flujo principal de resolución de producto. Es una feature avanzada de anuncios que no todos los clientes usan.
▾
Reordenar la resolución de producto
// classifier.ts / handler.ts // Orden actual: Redis → CTWA → keywords // Orden propuesto: Redis → keywords → CTWA (si habilitado) async function resolveProduct(msg, orgId, state) { // 1. Estado previo en Redis (siempre primero) if (state.productId) return state.productId; // 2. Keywords en texto (flujo normal) const byKeyword = await findProductByText(msg.text, orgId); if (byKeyword) return byKeyword.id; // 3. CTWA — solo si la org tiene referrals configurados if (msg.referral?.ctwa_clid && org.ctwa_enabled) { const byCTWA = await findProductByCTWA(msg.referral.ctwa_clid, orgId); if (byCTWA) return byCTWA.id; } // 4. Sin producto — usar default de org si existe return org.default_product_id ?? null; }
💡
Agregar ctwa_enabled: boolean a organizations permite activar/desactivar CTWA por cliente. Los que no usan anuncios no pagan el costo de esa query en cada mensaje.

P2 IMPORTANTE DB / BACKEND
Auto-crear campaña al crear un producto
El constraint de unicidad (org_id, product_id) ya garantiza una campaña por producto. Hay que aprovechar eso para generarla automáticamente.
▾
Opción A — Trigger en PostgreSQL
-- Trigger: al insertar un producto activo, crear campaña CREATE OR REPLACE FUNCTION auto_create_campaign() RETURNS TRIGGER AS $$ BEGIN INSERT INTO campaigns (organization_id, product_id, name, status) VALUES ( NEW.organization_id, NEW.id, 'Campaña de ' || NEW.name, CASE WHEN NEW.is_active THEN 'active' ELSE 'draft' END ) ON CONFLICT (organization_id, product_id) DO NOTHING; RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trg_product_campaign AFTER INSERT ON products FOR EACH ROW EXECUTE FUNCTION auto_create_campaign();
Opción B — En el endpoint POST /api/products (backend)
// Más explícito, más fácil de debuggear app.post('/api/products', async (c) => { const product = await createProduct(body, orgId); // Auto-crear campaña ligada (si no existe) await supabase.from('campaigns').upsert({ organization_id: orgId, product_id: product.id, name: `Campaña de ${product.name}`, status: product.is_active ? 'active' : 'draft', }, { onConflict: 'organization_id,product_id' }); return c.json(product); });
💡
Recomiendo la Opción B. Es más explícita, fácil de leer en el código y no requiere entender triggers de Postgres para mantenerla. El trigger es más "automático" pero oculta lógica de negocio importante.

▾
1
Guardar el ctwa_clid en el pago al validar comprobante

2
Llamar a la Conversions API de Meta al validar un pago

3
Agregar meta_pixel_id y meta_access_token a la org o instancia

Requiere que el cliente tenga un pixel de Meta y que los anuncios usen Click to WhatsApp (CTWA). No aplica si el cliente no hace ads en Facebook/Instagram.
