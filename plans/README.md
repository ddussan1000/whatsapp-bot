## Verification baseline (read before starting any plan)

This repo has **no automated test suite** (no `*.test.ts` in `backend/` or `dashboard/src/` at
commit `df009e2`) and **no test runner wired into `package.json`**. Verification gates in every
plan are therefore **typecheck + build + lint + a documented manual smoke test**, not unit tests.

- Backend typecheck: `cd backend && bun run check` → exits 0, no errors.
- Frontend typecheck + build: `cd dashboard && bun run build` → exits 0 (`tsc -b && vite build`).
- Frontend lint: `cd dashboard && bun run lint` → exits 0.

Do NOT introduce a test framework as part of these plans unless a plan explicitly says so.
If you believe one is warranted, STOP and report it as a separate recommendation.

## Out-of-band steps the executor cannot run

- **SQL migrations** (plan 001) live in `backend/scripts/sql/` and are run **manually by a human
  in Supabase Studio**. The executor writes the `.sql` file but must NOT attempt to apply it.
  Flag clearly in the handoff that a human must run it.
- **`bun run generate:api`** (regenerates `dashboard/src/lib/__gen__/api_v1.d.ts` from the live
  backend OpenAPI) requires a running backend on `localhost:3000`. Plans that add/modify routes
  note this as optional — the frontend code in these plans uses the hand-written wrappers in
  `dashboard/src/lib/api.ts` and types in `dashboard/src/types/api.ts`, not the generated file,
  so regeneration is not required for typecheck to pass.
